1. Contexto y objetivo
InvAR es una plataforma personal de análisis de inversiones para el mercado argentino. Su diferencial es un modelo de predicción que cruza señales del mundo político con movimientos de precios en BYMA, anticipando movimientos antes de que aparezcan en los datos financieros tradicionales.
FUNDAMENTO  
Argentina opera bajo una lógica donde los eventos políticos preceden a los movimientos financieros. Un decreto del ENRE sobre tarifas impacta en PAMP y CEPU antes de que cualquier analista lo refleje en un precio objetivo.
2. Regímenes políticos y estrategia de dataset
Argentina atravesó una ruptura estructural en diciembre 2023. El modelo de correlación evento-precio se invirtió en múltiples sectores. Por eso el dataset se divide en capas con pesos diferenciados.

ADVERTENCIA  
NO usar datos del kirchnerismo como base de entrenamiento. La relación evento-precio era opuesta: congelamiento de tarifas hacía subir a distribuidoras, acuerdo FMI hacía bajar bancos. Usarlos directamente confunde al modelo.
2.2 Volumen estimado de eventos por período
El dataset objetivo es de 600 a 900 eventos etiquetados para el entrenamiento inicial:
Fase consolidada (peso 1.0): ~400–500 eventos · fuente principal
Fase shock (peso 0.6): ~100–150 eventos · decretos y resoluciones grandes
Transición electoral (peso 0.4): ~50–80 eventos · solo electorales y macro
3. Estructura de base de datos — PostgreSQL
Todas las tablas viven en la base invar en PostgreSQL 15+. El esquema usa TIMESTAMPTZ para todos los timestamps (zona horaria America/Argentina/Buenos_Aires implícita en la app pero almacenada en UTC).

SQL de creación — eventos_politicos
CREATE TABLE eventos_politicos (
  id                BIGSERIAL PRIMARY KEY,
  fecha_evento      TIMESTAMPTZ NOT NULL,
  titulo            VARCHAR(500) NOT NULL,
  texto_completo    TEXT,
  fuente            VARCHAR(100) NOT NULL,
  url               VARCHAR(1000),
  tipo_evento       VARCHAR(30) NOT NULL,
  sector            VARCHAR(30) NOT NULL,
  sentiment_score   NUMERIC(4,3) NOT NULL CHECK (sentiment_score BETWEEN -1 AND 1),
  sentiment_fuente  VARCHAR(20) NOT NULL DEFAULT 'claude_api',
  regimen_politico  VARCHAR(30) NOT NULL,
  peso_entrenamiento NUMERIC(3,2) NOT NULL DEFAULT 1.0,
  tickers_afectados VARCHAR(20)[],
  verificado        BOOLEAN NOT NULL DEFAULT FALSE,
  creado_en         TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
-- Índices principales
CREATE INDEX idx_ep_fecha    ON eventos_politicos (fecha_evento DESC);

CREATE INDEX idx_ep_regimen  ON eventos_politicos (regimen_politico);
CREATE INDEX idx_ep_tipo     ON eventos_politicos (tipo_evento);
CREATE INDEX idx_ep_verif    ON eventos_politicos (verificado) WHERE verificado = FALSE;

CREATE TABLE impacto_precios (
  id              BIGSERIAL PRIMARY KEY,
  evento_id       BIGINT NOT NULL REFERENCES eventos_politicos(id),
  ticker          VARCHAR(20) NOT NULL,
  precio_t0       NUMERIC(18,4) NOT NULL,
  precio_t24      NUMERIC(18,4),
  precio_t48      NUMERIC(18,4),
  precio_t72      NUMERIC(18,4),
  var_pct_24h     NUMERIC(8,4) GENERATED ALWAYS AS
    (ROUND((precio_t24 - precio_t0) / precio_t0 * 100, 4)) STORED,
  var_pct_48h     NUMERIC(8,4) GENERATED ALWAYS AS
    (ROUND((precio_t48 - precio_t0) / precio_t0 * 100, 4)) STORED,
  var_pct_72h     NUMERIC(8,4) GENERATED ALWAYS AS
    (ROUND((precio_t72 - precio_t0) / precio_t0 * 100, 4)) STORED,
  etiqueta        VARCHAR(10) GENERATED ALWAYS AS (
    CASE WHEN (precio_t48-precio_t0)/precio_t0*100 > 2  THEN 'SUBE'
         WHEN (precio_t48-precio_t0)/precio_t0*100 < -2 THEN 'BAJA'
         ELSE 'NEUTRO' END) STORED,
  merval_var_48h  NUMERIC(8,4),
  creado_en       TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  UNIQUE(evento_id, ticker)
);
NOTA ML  
var_pct_48h es la variable objetivo del modelo: lo que el ML aprende a predecir. etiqueta es la versión categorizada (SUBE/BAJA/NEUTRO) para clasificación binaria.

5. Flujo de ingesta y etiquetado
5.1 Pipeline de un nuevo evento en tiempo real
1. 
Scraper detecta nuevo evento en fuente (ENRE, BCRA, Twitter oficial, BOE)
2. 
Node.js inserta fila en eventos_politicos con verificado = FALSE
3. 
Job llama a Claude API: extrae sector, tipo y sentiment_score
4. 
Se actualizan los campos clasificados. Queda pendiente verificación manual
5. 
A t+48h, job automático busca el precio del ticker en cotizaciones y completa impacto_precios
6. 
Si existe predicción activa, se completa resultado_real y acerto
7. 
Registro disponible para próximo ciclo de reentrenamiento
5.2 Pipeline del dataset histórico (etiquetado manual inicial)
Para construir el dataset base de jul 2024 → abr 2026 se necesita:
Exportar archivo CSV de noticias históricas desde Ámbito, Infobae y fuentes oficiales
Correr script Python que llama a Claude API en batch para pre-clasificar sector y sentiment
Revisor humano valida y corrige en interfaz web simple (campo verificado = TRUE)
Script cruza automáticamente con histórico de precios de BYMA para poblar impacto_precios
Dataset listo para entrenamiento con scikit-learn (Random Forest o XGBoost)
ESTIMACIÓN  
Se estima 3–4 horas de revisión manual para verificar ~500 eventos pre-clasificados por Claude API. La precisión de Claude en clasificación de sector es ~92% en pruebas iniciales, reduciendo el trabajo manual al ~8% de correcciones.

8. Checklist de implementación para Antigravity
INSTRUCCIÓN  
Este checklist define el orden de construcción recomendado. Cada ítem es independiente y puede asignarse a un agente distinto.
Fase 1 — Base de datos (prioridad máxima)
Crear base de datos invar en PostgreSQL 15
Ejecutar schema.sql existente (cotizaciones, señales, noticias, portafolio)
Crear tabla eventos_politicos con todos los campos e índices del §3.1
Crear tabla impacto_precios con columnas generadas del §3.2
Crear tabla predicciones_ml del §3.3
Verificar constraints: CHECK en sentiment_score, UNIQUE en impacto_precios(evento_id, ticker)
Fase 2 — Ingesta histórica
Script Python: leer CSV de noticias históricas jul 2024 → abr 2026
Llamada batch a Claude API para clasificar sector, tipo y sentiment de cada fila
Insertar resultados en eventos_politicos con verificado = FALSE
Script de cruce: para cada evento con tickers_afectados, buscar precio en cotizaciones y poblar impacto_precios
Interfaz de revisión manual: listar eventos con verificado = FALSE, formulario de corrección
Fase 3 — Modelo ML
Script Python: exportar dataset de entrenamiento desde vista ml_dataset (eventos + impacto con peso)
Entrenar Random Forest con scikit-learn usando features del §4
Serializar modelo con joblib (modelo_v1.pkl)
Endpoint Node.js POST /api/predict: recibe evento nuevo, llama a Python via child_process, devuelve predicción
Job cron: a t+48h busca resultado real, completa impacto_precios y predicciones_ml
Fase 4 — Frontend
Columna nueva en módulo Análisis: "Señal política" con prob % y ventana temporal
Panel de predicciones activas: ticker, señal, confianza, tiempo restante hasta vencimiento
Historial de aciertos por sector y modelo_version
9. Vista auxiliar: ml_dataset
Esta vista facilita la exportación del dataset para entrenamiento sin joins manuales:
CREATE OR REPLACE VIEW ml_dataset AS
SELECT
  ep.id,
  ep.fecha_evento,
  ep.tipo_evento,
  ep.sector,
  ep.sentiment_score,
  ep.regimen_politico,
  ep.peso_entrenamiento,
  EXTRACT(HOUR FROM ep.fecha_evento) AS hora_evento,
  ep.fuente,
  ip.ticker,
  ip.precio_t0,
  ip.var_pct_24h,
  ip.var_pct_48h,
  ip.var_pct_72h,
  ip.etiqueta,
  ip.merval_var_48h
FROM eventos_politicos ep
JOIN impacto_precios ip ON ip.evento_id = ep.id
WHERE ep.verificado = TRUE
  AND ep.peso_entrenamiento > 0
ORDER BY ep.fecha_evento DESC;
